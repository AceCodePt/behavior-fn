How It Works

The auto-loader pattern automatically adds is="behavioral-*" attributes to elements with behavior attributes. This gives you cleaner HTML at the cost of some performance overhead.

Step 1: Load Core Runtime

<script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/behavior-fn-core.js"></script>

Same as the explicit pattern.

Step 2: Load Behaviors You Need

<script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/reveal.js"></script>

Same as the explicit pattern.

Step 3: Load Auto-Loader

<script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/auto-loader.js"></script>

That's it! The auto-loader automatically enables itself when loaded. No extra JavaScript needed.

Step 4: Omit is Attributes

<!-- No is attribute needed! -->
<dialog behavior="reveal" id="my-modal">
  Content here
</dialog>

<button commandfor="my-modal" command="--toggle">
  Open Modal
</button>

The auto-loader automatically adds is="behavioral-reveal" when it detects behavior="reveal".

<!DOCTYPE html>

  <meta charset="UTF-8">
  <title>Auto-Loader Pattern Example</title>
  
  <!-- 1. Load core runtime -->
  <script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/behavior-fn-core.js"></script>
  
  <!-- 2. Load behaviors -->
  <script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/reveal.js"></script>
  <script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/logger.js"></script>
  
  <!-- 3. Load auto-loader (auto-enables itself) -->
  <script src="https://unpkg.com/behavior-fn@0.2.0/dist/cdn/auto-loader.js"></script>
</head>
<body>
  <!-- 4. Omit is attributes (auto-loader adds them) -->
  <dialog behavior="reveal" id="my-modal">
    <h2>Hello!</h2>
    <button commandfor="my-modal" command="--hide">Close</button>
  </dialog>
  
  <button commandfor="my-modal" command="--toggle">
    Open Modal
  </button>
</body>
</html>

Tradeoffs

Aspect Explicit Pattern Auto-Loader Pattern
HTML More verbose (is attribute) Cleaner (no is attribute)
Bundle Size Smaller (no auto-loader) +5.4KB
Performance Best (no observer) MutationObserver overhead
Debugging Easy (explicit) Harder (implicit)
Dynamic Content Manual is attributes Automatic detection

When to Use Auto-Loader?

  • Prototypes: Speed of development > performance
  • Content sites: Lots of dynamic content
  • CMS integrations: Can't control HTML output
  • Quick demos: Cleaner HTML for examples

When NOT to Use Auto-Loader?

  • Production apps: Use explicit pattern for best performance
  • Performance-critical: Every KB matters
  • Large DOM: MutationObserver overhead increases with DOM size

Other Patterns

  • Explicit Pattern - Recommended for production
  • ESM Pattern - For bundlers
